## 完整正文备用

这条讲 Codex CLI 的权限安全：`--sandbox` 三种模式怎么理解，`--ask-for-approval` 什么时候用 `on-request` 或 `never`，以及交互会话里怎么用 `/permissions`、`/status`、`/debug-config` 核对当前权限。演示用脱敏测试目录，不展示真实 Key 或真实项目。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
看到 `--sandbox`、`--ask-for-approval`、`/permissions` 时不知道区别
不清楚什么时候可以让 Codex 自动跑，什么时候必须人工看一眼
容易一上来开 `danger-full-access`，但没有隔离目录、Git 状态和备份
看到审批提示只看“允许/拒绝”，没有检查命令、路径、网络、Key、`.env`

看完这篇，你应该能做到：
用 `codex --help` 确认本机 Codex 支持的权限参数
区分 `read-only`、`workspace-write`、`danger-full-access` 的适用场景
区分 `untrusted`、`on-request`、`never` 的审批策略边界
在交互会话里用 `/permissions` 调整权限，用 `/status` 核对当前模型、审批策略、可写目录和上下文信息

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

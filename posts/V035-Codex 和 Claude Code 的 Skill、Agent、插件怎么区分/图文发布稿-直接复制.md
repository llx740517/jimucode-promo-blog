# Codex 和 Claude Code 的 Skill、Agent、插件区别讲清楚

这条用真实终端入口和脱敏示例，把 Codex / Claude Code 里容易混的 Skill、Agent、插件拆开讲。重点看三件事：什么场景该沉淀成 Skill，什么场景该交给 Agent，什么时候才需要插件打包分发。视频里会展示 `codex plugin`、`claude agents`、`claude plugin` 等入口，并说明哪些内容必须录屏前实测确认。

这篇适合刚开始接触Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按这条顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

(1)常见卡点

把 Skill 当成 Agent，结果只写了一段提示词，却期待它自己跑任务；把 Agent 当成插件，结果没有分发和安装机制，换项目就失效；把插件当成 MCP，结果不知道插件是包装层，MCP 是外部工具协议；只看功能名，不看入口、文件位置、权限和调试方式。

(2)看完这篇，你应该能做到

用一句话判断当前需求更像 Skill、Agent 还是插件；分别知道 Codex 和 Claude Code 哪些入口可以录屏确认。

知道 Claude Code 子代理的典型文件位置：`.claude/agents/`、`~/.claude/agents/`、插件 `agents/` 目录，以及 `--agents` 临时 JSON

知道插件不是“万能 AI”，而是安装、启用、管理一组能力的包装。

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

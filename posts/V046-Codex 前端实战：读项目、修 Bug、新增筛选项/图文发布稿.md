## 完整正文备用

这条用一个脱敏前端演示项目录 Codex 实战：先读项目结构，再修一个按钮无响应 Bug，再新增一个筛选项，最后看 `git diff`、浏览器验证，并运行构建或测试。重点不是让 AI 一次改完所有问题，而是把前端任务拆小、每步可验证。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
只会问“帮我改一下”，不会先让 Codex 读项目结构和运行方式
前端 Bug、筛选项、状态展示混在一起提，导致改动范围太大
改完不看浏览器、不看 diff、不跑测试或构建
演示项目包含真实接口、真实用户信息或内部页面，录屏风险高

看完这篇，你应该能做到：
用 Codex 进入一个前端项目，先读 `package.json`、入口文件、页面组件和状态来源
设计一个可录屏的小任务链：修按钮无响应 Bug，再新增一个状态筛选项
让 Codex 小步修改，并用 Git diff 检查改动范围
用浏览器页面、终端命令和测试/构建输出确认结果

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

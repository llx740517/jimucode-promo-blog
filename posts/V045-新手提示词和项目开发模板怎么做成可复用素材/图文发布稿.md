## 完整正文备用

这条演示怎么把 Bug 修复、新功能、代码审查、发布说明这几类常用提问做成项目里的 Markdown 模板。录屏会分别展示 Codex 和 Claude Code 的入口差异，并提醒哪些内容可以复用，哪些变量必须每次重新填写和人工核对。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
提问靠临场发挥，同一个问题每次写法都不一样，AI 输出不稳定
只收藏“10 条提示词”，但没有变量、验收标准和复用方式，真正开发时仍然不知道怎么填
Codex 和 Claude Code 的命令入口、项目规则文件不一样，容易把两个工具的用法混在一起
模板里容易塞进真实业务日志、Key、用户信息或私有路径，录屏和图文发布时有泄露风险

看完这篇，你应该能做到：
把提示词拆成“模板固定部分”和“每次手填变量”
建一个演示用的 `prompts/` 模板目录，至少包含 Bug 修复、新功能、代码审查、发布说明四类模板
在 Codex 中用 `-C` 指定项目目录，用 `codex exec` 或 `codex review --uncommitted` 处理模板化任务
在 Claude Code 中用 `CLAUDE.md` 保存项目长期规则，用 `--append-system-prompt-file` 或 `--system-prompt-file` 加载单次模板

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

## 完整正文备用

这条视频不讲安装，也不把 Hooks 说成万能自动化。重点演示 Hooks 在生命周期里什么时候触发、适合做哪些日志和检查、Claude Code 怎么看 debug 文件，以及 Codex hooks 目前哪些内容需要录屏前实测确认。适合已经能正常使用 Codex / Claude Code，想继续做自动化和故障排查的人。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
只知道“Hooks 可以自动执行脚本”，但不清楚它在生命周期里什么时候触发
录屏时容易把 Hooks 讲成万能自动化，忽略权限、日志、调试和误触发风险
Claude Code Hooks 资料较完整，Codex 本地资料只确认到 hook trust 和 managed hooks 相关内容，不能把两者混成同一套配置
自动化失败时不知道先看终端、配置、Key/API 地址、权益状态、日志页、网络、权限还是项目目录

看完这篇，你应该能做到：
看懂 Hooks 在“用户提交、工具调用前后、结束、配置变更、失败”这些节点上能做什么
能设计一个安全的录屏演示：用 Claude Code 的 `PostToolUse` 或 `Stop` hook 记录脱敏日志，并展示 debug 文件如何辅助排查
能区分 Codex 和 Claude Code 的边界：Claude Code 展示可确认的 hooks 设置和调试方式；Codex 只展示本地已确认的 hook trust / managed hooks 线索，其余写“需录屏前实测确认”

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

# GitHub、数据库、浏览器、Playwright MCP 适合做什么

这条用 GitHub、数据库、浏览器、Playwright 四类场景，讲清楚 MCP 适合扩展哪些能力、录屏前要准备哪些 Token 和测试环境、Codex 与 Claude Code 的命令入口有什么差异，以及连接失败时先看哪些位置。重点是场景判断、权限边界和打码方案，不承诺每个第三方服务都能当场跑通。

这篇适合刚开始接触Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按这条顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

(1)常见卡点

把 MCP 理解成“接上就自动完成所有事”，导致录屏时容易翻车；分不清 GitHub MCP、数据库 MCP、浏览器 MCP、Playwright MCP 的适用场景；不知道 Codex 和 Claude Code 的 MCP 命令、配置方式、验证方式有什么差异；连接失败时只反复重试，不知道应该按终端输出、配置文件、Key/API 地址、权益状态、日志页、网络、权限、项目目录的顺序排查。

(2)看完这篇，你应该能做到

用场景判断 MCP 是否值得接，而不是先盲目安装；区分 GitHub、数据库、浏览器、Playwright 四类 MCP 的价值和风险；看懂 Codex 与 Claude Code 在 MCP 管理命令、配置方式、鉴权方式上的差异；录屏前知道哪些 Token、数据库、项目、页面必须先准备，哪些内容需要打码。

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

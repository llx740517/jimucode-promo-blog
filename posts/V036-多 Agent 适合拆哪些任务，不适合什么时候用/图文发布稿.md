## 完整正文备用

这条用一个小项目演示多 Agent 的正确使用边界：哪些任务适合拆成读代码、查问题、写测试、复核，哪些任务不适合拆。Claude Code 侧会讲 `--agent`、`--agents`、`claude agents` 和 `.claude/agents/`；Codex 侧只讲已确认的 `multi_agent` 特性和需录屏前实测确认的入口，不把两套命令混在一起。最后用 `git diff` 和测试结果收口。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
把多 Agent 当成万能加速器，简单任务也硬拆，反而更慢
不知道哪些任务能并行，哪些任务必须由一个主线上下文统一决策
混淆 Codex 和 Claude Code 的 Agent 命令、配置和入口
Agent 做完后没有汇总、diff、测试和边界检查

看完这篇，你应该能做到：
用“能否独立调查、能否独立验证、是否共享同一上下文”判断要不要拆 Agent
把一个小项目任务拆成读代码、查问题、写测试、复核风险四类角色
区分 Claude Code 可确认的 `--agent`、`--agents`、`claude agents`、`.claude/agents/`，以及 Codex 侧需录屏前实测确认的多 Agent 入口
认识不适合多 Agent 的场景：一行小改、强耦合设计、涉及密钥和生产权限、没有验证命令的任务

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

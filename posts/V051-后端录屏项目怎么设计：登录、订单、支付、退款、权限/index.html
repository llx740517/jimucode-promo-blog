完整正文备用

这条讲后端实战录屏前怎么设计演示项目：用假登录、假订单、mock 支付、假退款单和权限矩阵，准备可复现接口、日志和测试命令。重点是不直接录真实生产系统，先把素材做成可控、脱敏、可验证的后端 demo，服务后续 Codex / Claude Code 实战视频。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点： 真实后端项目里有数据库、Token、用户数据、订单号、支付回调，录屏风险高 后端实战一旦没有设计好，AI 排查会变成随机看日志，观众看不懂 登录、订单、支付、退款、权限混在一起，任务太大，无法稳定复现和验证 不知道哪些画面适合作为后续 V049/V050 的母素材，哪些必须避免拍

看完这篇，你应该能做到： 设计一个可录屏的脱敏后端 demo 项目，而不是直接录真实生产系统 把后端实战拆成五段：登录、订单、支付、退款、权限 给每段预设可复现现象、接口请求、日志、测试命令和可解释边界 明确哪些数据必须用假值：用户、订单、支付流水、退款单、权限角色、数据库连接

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

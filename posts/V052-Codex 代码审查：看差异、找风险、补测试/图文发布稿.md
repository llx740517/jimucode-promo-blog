## 完整正文备用

这条用一个脱敏小 PR 演示 Codex 代码审查流程：先用 `git diff` 看改动范围，再用 `codex review --uncommitted` 和 `codex review --base main` 找风险，接着围绕风险补最小测试，最后跑验证并整理 PR 描述和上线检查清单。

重点不是把审查交给 AI，而是把 Codex 放进一个可重复的开发流程里。

这篇适合刚开始接触积木代码助手、Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按图里的顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

常见卡点：
AI 改完代码后说“完成了”，但用户不知道到底改了哪里
本地有 staged、unstaged、untracked 改动，审查范围不清楚
PR 分支和 `main` 的差异较多，不知道先看哪几处风险
测试失败后容易让 AI 大范围重写模块，反而引入更多变动

看完这篇，你应该能做到：
演示一个同一 PR/分支里的完整审查路线
展示 `codex review --uncommitted` 和 `codex review --base main` 的使用场景
讲清楚读 review 输出时先看哪些信号：风险、边界条件、缺测试、验证建议
给出补测试时的限制方式：只围绕失败点或风险点，不重写整个模块

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。

# Claude Code 后端实战：接口 500 怎么读日志、定位原因、补测试

这条用一个脱敏 FastAPI 演示项目，录 Claude Code 从接口 500 到补测试的完整流程：先复现问题，再看日志和测试，让 Claude Code 先给计划，小步修改后看 diff，最后跑测试和接口请求确认结果。重点提醒后端项目里的 Key、数据库地址、订单号、支付信息和用户数据必须打码。

这篇适合刚开始接触Codex 或 Claude Code 的同学。不要只盯着一个按钮或一条命令，建议按这条顺序看：先看当前问题，再看操作路径，最后确认结果有没有真正跑通。

(1)常见卡点

接口报 500 时只把错误截图发给 AI，缺少路由、请求参数、日志和测试上下文；让 AI 直接改代码，最后不知道它改了哪里，也不知道是否影响登录、订单、支付或退款逻辑；后端项目里容易出现数据库地址、订单号、支付回调密钥、用户信息，录屏时有泄露风险；Claude Code 的权限模式、项目记忆文件和工具限制没有提前设置，录制中容易跳权限提示或误改文件。

(2)看完这篇，你应该能做到

用一个可控的后端演示项目复现接口 500；让 Claude Code 先读接口结构、日志和测试，再提出定位计划；让 AI 小步修改后端代码，而不是一次重写整个模块；用 `git diff`、`pytest -q`、`curl -i` 或 API 客户端确认结果。

我的建议是，第一次操作时不要一边改很多地方，一边猜原因。先把页面、终端输出、配置文件、日志记录这几块分开看，哪一步不一致，就从那一步往回查。

如果你也在配置或使用 AI 编程工具，可以先收藏这篇。后面遇到类似问题时，按这条路线重新核对一遍，通常能更快判断下一步该看哪里。
